1. 产品目标

Label 平台的目标是把已有 ZPL 标签资产转化为可管理、可测试、可批量生成、可通过 API 调用、可按打印规则派发、可追踪监控的企业级标签服务。

当前最小版本优先支持“导入已有 ZPL 并二次加工”，全新绘制标签暂不作为 MVP 范围。

2. 用户角色

角色主要诉求典型操作
业务系统调用方 通过统一 API 生成 PDF、ZPL 或打印任务。 调用 Print Service REST API，传入 JSON/XML 业务数据。
业务/运营人员 导入模板、确认字段、测试 PDF、批量生成标签。 上传 ZPL、确认字段来源、下载测试 PDF、查看生成结果。
仓库/IT 管理员 维护打印机、路由规则、打印队列和状态。 配置打印机、绑定模板、设置路由、重试或补打。
开发/支持人员 调试 API、排查错误、查看日志和 Dashboard。 生成请求示例、运行 API 测试、按 request ID 查日志。

3. 系统组件总览

客户端

面向运营、IT 和开发人员的浏览器管理台，负责模板制作、字段确认、API 测试、打印配置、队列恢复和日志监控。

服务端

面向业务系统和打印执行的服务进程，负责 REST API、渲染、队列、打印服务器长连接、设备状态采集和直连 IP 网关。

Label 制作

导入已有 ZPL，保留原始快照，创建工作副本，支持预览、轻量编辑、版本管理和测试样本。

数据源处理

解析 ZPL 字段，生成字段确认表单，人工确认字段来源，保存可复用数据源配置，支持批量生成。

API 接口封装测试

提供 REST API、API 调试页面、请求示例生成、鉴权、参数校验和统一错误码。

Print Service

业务系统打印入口层，负责 JSON/XML 转换、字段映射验证、模板和打印机候选选择。

打印配置网关

维护打印服务器、打印机配置、模板绑定、路由规则、打印队列、重试、暂停、取消和补打。

接口日志监控

记录 API、渲染、打印日志，提供错误详情、重试记录、筛选和 Dashboard。

4. 核心业务流程

4.1 导入已有 ZPL 并创建模板

  1. 用户上传或粘贴已有 ZPL 文件。
  2. 系统保存原始 ZPL 作为 source snapshot。
  3. 系统创建 working copy 作为后续二次加工版本。
  4. 使用 labelize 解析和渲染，生成 PNG/PDF 预览。
  5. 用户保存模板元数据：名称、尺寸、DPI/DPMM、描述、版本。

4.2 字段确认与数据源配置复用

  1. 系统使用 labelize 解析 ZPL，抽取文本、条码、二维码/DataMatrix 等候选字段。
  2. 系统生成格式化字段表单，包含字段名、当前值、元素类型、位置和建议来源。
  3. 用户确认字段来源：manual、api_field、fixed、ignored。
  4. 系统保存字段映射、校验规则、默认值、转换规则和批量导入设置。
  5. 同一客户、供应商、仓库或业务流程可复用已保存配置。

4.3 API 测试生成 PDF

  1. 业务或技术用户选择模板和数据源配置。
  2. 系统根据确认字段自动生成请求 JSON。
  3. 用户 POST 测试数据。
  4. 系统校验字段、合并数据、调用 labelize 渲染。
  5. 系统生成 PDF，返回下载/预览地址，供业务人员确认。

4.4 业务系统打印请求

Business System -> Print Service REST API -> JSON/XML normalization -> field mapping and validation -> template selection -> printer or route candidate selection -> Print Configuration Gateway -> route rule resolution -> print queue -> Print Server long connection OR Direct Printer IP Gateway -> Printer Device / QZ Tray / PDF only

5. 模块规格

5.1 Label 制作

能力规格
创建方式MVP 支持导入已有 ZPL；未来支持从空白画布全新绘制。
版本管理保留原始 ZPL；每次编辑保存为新模板版本。
预览通过 labelize 输出 PNG/PDF；300dpi 标签使用 12 dpmm。
测试样本每个模板可保存样本数据和预期输出，用于回归验证。

5.2 数据源处理

能力规格
字段抽取从 ZPL 的文本、条码和二维代码内容中抽取候选字段。
字段来源manualapi_fieldfixedignored
配置复用按模板、客户、供应商、仓库、业务流程保存和复用配置。
批量生成支持 JSON、CSV、Excel、API 结果；行级校验和部分失败。

5.3 API 接口封装测试

能力规格
API 调试页面选择模板和数据源配置，自动生成请求体，运行测试，展示响应和预览。
请求示例生成 cURL、JavaScript fetch、Python requests 和 raw JSON。
输出类型png、pdf、zpl、print_task、pdf_test。
鉴权API Key，使用 Authorization: Bearer <api_key>，支持 scope。
错误规范统一错误格式，所有错误包含 request_id 和稳定错误码。

5.4 Print Service

职责说明
RESTful 入口接收业务系统发出的打印请求。
数据转换JSON 优先，后续支持 XML 转换为统一字段模型。
字段验证校验必填、类型、长度、枚举、业务规则。
模板选择根据请求参数、路由上下文或默认规则选择模板。
打印机候选选择根据仓库、站点、业务类型、客户、供应商、模板选择候选打印机或路由。
边界不直接保存底层连接、不调度队列、不直接向设备发送 ZPL。

5.5 打印配置网关

打印队列高优先级要求 API 请求处理、数据映射和任务创建必须与物理设备派发解耦。即使单位时间内出现批量打印请求，或打印机无响应，服务进程也必须保持可响应；界面必须用醒目的红/橙色提示暴露积压、失败、离线和待重试任务，并支持一键筛选到受影响队列。
能力规格
客户端/服务端边界客户端只负责配置、监控和人工操作；服务端负责打印服务器连接、队列调度、设备状态采集和物理打印派发。
打印服务器连接服务端可连接 Windows Print Server、CUPS、厂商网关或自定义 HTTP 网关，并保持长连接/心跳。
打印服务器状态同步服务端周期读取打印机在线状态、错误状态、纸张/耗材状态、位置、队列状态和最近心跳。
直连 IP 网关没有打印服务器时，服务端可直接连接打印机 IP/端口，发送原始 ZPL/EPL，并在超时或无响应时进入重试/待人工处理队列。
打印机配置连接类型、打印服务器 ID、打印服务器设备名、IP、端口、DPI/DPMM、纸张尺寸、型号、位置、站点、仓库。
模板绑定模板可配置允许打印机和默认打印机，可覆盖纸张、DPI、份数。
路由规则按仓库、站点、业务类型、模板、客户、供应商、优先级匹配。
队列状态queued、dispatching、sent、completed、failed、paused、canceled。
服务响应打印任务入队后异步派发，物理打印机离线或超时不能阻塞 API 进程。
队列告警按队列深度、最老等待时间、失败/阻塞任务、重试次数和离线设备生成 warning/critical 提示。
人工恢复告警卡片可一键进入过滤后的任务列表，支持重试、改派、暂停、取消和补打。
操作重试、暂停、恢复、取消、补打。
QZ Tray作为可选本地工作站打印通道，与直接网络打印并列。

5.6 接口日志监控

能力规格
API 请求日志记录 method、path、endpoint、调用方、API key、状态码、耗时、错误码。
渲染日志记录模板、版本、输出类型、DPMM、耗时、输出路径、错误详情。
打印日志记录打印任务状态、选中打印机、路由规则、通道、重试和 attempt。
筛选按模板、接口、打印机、仓库、站点、业务类型、时间、状态、错误码。
Dashboard成功率、失败率、平均耗时、p95 耗时、调用量、队列深度、失败任务数、打印队列健康状态。
醒目提示Dashboard 顶部展示队列告警卡片，包含严重级别、影响数量、建议动作和任务筛选入口。

6. API 摘要

接口用途主要返回
POST /api/v1/labels/render生成 PNG/PDF/ZPL。request_id、output_url
POST /api/v1/labels/test-pdf业务测试用 PDF 标签。request_id、PDF output_url
POST /api/v1/labels/printPDF 预览或设备打印。output_url 或 print_task_id
POST /api/v1/labels/batch-render批量生成标签。batch_job_id、行级统计
GET /api/v1/data-source-configs搜索数据源配置。配置列表
GET /api/v1/printers搜索打印机。打印机列表
GET /api/v1/print-tasks查询打印任务。任务列表
GET /api/v1/dashboard/summary监控总览。成功率、失败率、调用量、队列深度、print_queue_health

7. 核心数据对象

Template

模板 ID、名称、格式、原始内容、工作内容、尺寸、DPMM、版本、状态。

Template Field

字段名、显示名、来源类型、元素类型、原始值、默认值、必填、位置、格式规则。

Data Source Config

模板、客户、供应商、仓库、业务流程、映射规则、校验规则、默认值、转换规则。

Printer

名称、站点、仓库、连接类型、IP、端口、QZ 客户端、本地打印机名、型号、DPMM、纸张。

Print Task

request ID、模板、打印机、路由规则、通道、payload、份数、状态、重试次数、时间戳。

Logs

API Request Log、Render Log、Error Log、Print Task Attempt。

9. 开发路线图

  1. Render Service Foundation：模板存储、ZPL 导入、PDF/PNG 预览、请求日志、基础测试。
  2. Template And Data Management：字段抽取、字段确认、数据源配置管理、批量生成。
  3. API Integration Workbench：API 调试台、请求示例、API Key、统一校验和错误码。
  4. Print Gateway：打印机配置、模板绑定、路由规则、QZ Tray、队列和重试。
  5. Monitoring Console：日志筛选、错误详情、重试记录、Dashboard。
  6. Advanced Designer：全新绘制、半可视化编辑、字段提示、视觉 diff。
